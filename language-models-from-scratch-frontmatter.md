# 从零理解语言模型

**序**

我习惯把语言模型当成一个软件系统来读。这种习惯不是从论文里来的，是被几次线上事故教出来的。

最早一次是几年前。我们做了一个内部资料问答助手——上传产品文档、设计稿、会议纪要，用户可以问"这个功能下个版本还在吗"。前两周很顺利。回答流畅，有引用，有摘要，看起来像个聪明的同事。

直到一位产品经理问了一句很普通的话："批量导入的旧格式还能用多久？"助手回答："下个季度继续支持，没有迁移风险。"语气稳，引用有，三条理由。那位 PM 把这话写进了给客户的邮件。

旧格式其实已经计划在下个小版本废弃。

第一反应当然是：模型幻觉了。这话也没错，但太粗——跟把所有线上事故都叫"系统坏了"是一个层级。如果停在这一步，下一步通常就是换更大的模型、写更长的 prompt、让模型"更谨慎一点"。这些办法有时候有用，更多时候只是把错误藏得更深一点。

后来真正排查下来，问题不在模型本身。是整条链路上每一环都做了一点合理的事，合起来出了事。

检索取回了两份文档。一份是去年的客户说明，写着"下个季度继续支持"；一份是上个月的迁移计划，写着"下个小版本废弃"。两份都被取出来了，但旧文档排在前面——因为它的标题更贴近用户的问法，而新文档的标题里没有"废弃"两个字。再往下看，文档元数据也乱：旧客户说明根本没标时间戳，新迁移计划的状态字段是空的。检索器只看表面相似度，于是旧文档分高。模型不是故意忽略新资料，它就是在一个已经偏斜的上下文里生成了一段看起来一致的回答。

prompt 写的是"根据资料回答并引用来源"。它没说遇到冲突要先报警。

评估集也没覆盖这种情况。我们测了事实问答、摘要、引用格式、拒答——就是没测"同一问题存在新旧两份冲突文档"。上线前的分数很好看，因为压根没人问过这种问题。

监控只存了模型的最终输出。检索排序、文档版本、每条引用对应原文的哪一段，全都没记。事故发生以后，我们花了一整天才复现当时模型到底看到了什么。

最后的修复一件都和"换更强的模型"无关。我们做了几件很无聊的事：文档进系统前必须带时间戳、来源类型、版本状态、所有者；检索排序加入新鲜度和来源权威性，旧文档不是不能用，但不能在跟新迁移计划冲突时占默认位；prompt 增加冲突处理——遇到矛盾证据，模型必须先说明冲突再给结论；评估集加上冲突样本、过期样本、缺失样本、错误前提样本；日志从"保存最终答案"扩展成"保存完整决策链"，下一次出错时至少有得查。

这件事很小，却几乎触到了语言模型工程的全部主题。Tokenizer 决定文档怎么进模型，context 决定能放多少证据，embedding 和检索决定哪些片段会出现，prompt 决定模型怎么用，权重决定它生成什么，推理参数决定是否稳定，评估决定上线前能不能发现，日志决定事故后能不能复盘，权限和流程决定错误答案会不会直接砸到客户。

如果只把它叫"幻觉"，这些环节就全消失了。如果把它当成一次系统事故，改进路径就清楚得多。

---

这本书就是从这个视角写的。

我不会把语言模型写成神秘智能，也不打算把它降格成一个普通 API。我想说的是：语言模型是一类由数据、计算、训练目标、推理系统、工具、评估和组织流程共同组成的工程系统。模型权重只是最显眼的那块，不是全部。

主线大致是这样：

```text
文本 → token → 张量 → Transformer → 训练 → 数据 → 推理 → 评估 → 对齐 → 系统设计
```

这条线看起来像目录，其实更像故障排查路径。事实错误可能来自数据、检索、上下文或评估；格式错误可能来自 SFT、schema、解码参数或 parser；延迟问题可能来自 prefill、decode、KV cache 或调度；安全问题更多时候是权限、工具边界、日志的事，未必是模型本身的事。

写法上我没什么野心。少用定义开场，多用具体失败开场；公式只在它真的压缩了工程事实的时候才出现；少写"模型将改变一切"，多写"这个选择会增加多少 token、多少显存、多少延迟、多少看不见的风险"。如果一定要总结一句风格目标，那就是：把结构讲清楚，把抽象讲稳，把人的直觉留在场，把句子写到能连续读下去。

读这本书我推荐一种读法——每章问四个问题：这一章处理的是系统里的哪一环？它保存了什么状态？它怎么消耗计算和数据？如果它出错，错误怎么传到用户面前？这四个问题答得上来，新模型、新工具、新论文就不会只是新闻。你能把它们放回这条链路里看。

谁适合读？我心里大概想着四类人。一种是对语言模型好奇但不想被术语淹没的爱好者；一种是已经会用 API、希望理解黑盒下面发生了什么的开发者；一种是学过机器学习、但缺一张 LLM 工程全局图的学生；还有一种是想进 AI infra、应用工程或 research engineering 的工程师。你不需要一上来就懂 CUDA、分布式训练或强化学习，但你需要愿意把一个看起来神奇的系统拆成普通部件。

剩下的话留给正文。我们从一个最小的问题开始：为什么要从零理解一个能直接用 API 调到的东西。
